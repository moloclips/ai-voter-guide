You are reviewing a candidate for the AI Voter Guide project.

Workflow
1. Review the candidate's current evidence.
2. Check whether any current evidence needs revision or deletion.
3. Search the web for additional direct evidence about this candidate's AI-related record.
4. Based on the revised evidence, decide whether Status or Verdict should be modified.

Working constraints:
- use the provided candidate JSON, evidence JSON, prior changes JSON, and web search as your primary inputs
- do not inspect unrelated local files
- do not inspect `AGENTS.md`, `SKILL.md`, or other repo instruction files unless strictly necessary
- do not run shell commands unless strictly necessary

Things to check for:
- New evidence
	- candidate dropped out or lost a primary
	- direct campaign, government, bill, interview, debate, or press release sources.
- Existing evidence
	- use the exact Evidence_ID as the key for mod or del
	- Ballotpedia election-page URLs should usually be replaced with more direct sources
	- Ballotpedia candidate-specific pages may be deleted only if a direct source already exists or is being added in the same batch
	- soft-404
	- dead link
	- irrelevant to candidate
	- misleading description
	- long description. Usually aim for 5-14 words and format as "Description (Source)"; use more words when needed to briefly explain what a bill, vote, or proposal does and why it matters for the candidate's AI record
	- assume an uninformed voter reading the description without prior knowledge of the bill name, acronym, or legislative context; prefer descriptions that convey what the legislation or statement does, not just its title or a generic AI label
	- the source name should appear parenthetically at the end of the description
	- for official sources, prefer concise standardized parenthetical source labels like "(House Rep Website)", "(Senator Website)", "(Governor Website)", or "(Campaign Website)"
	- avoid negative assertions like "No AI position found" in evidence descriptions; the verdict should capture absence of an AI record
	- if a source is still worth keeping but contains no AI-specific content, prefer a neutral description with one of those parenthetical source labels
	- before proposing any mod to `Source_Description`, inspect the current source itself, not just the existing description
	- only rewrite a description if the new wording is directly supported by the source as currently loaded
	- do not shorten a description if doing so would remove the key AI-related claim
	- if the source contains a specific AI-related statement, action, vote, bill, or proposal, prefer a specific claim-focused description over a generic site label
	- if the source is weak, irrelevant, dead, soft-404, or does not support the rewritten claim, prefer del or replacement over cosmetic shortening
	- when revising an existing evidence description, preserve the most important supported policy angle from the current description unless the source clearly does not support it; when multiple concise descriptions are supported, choose the one most relevant to an AI single-issue voter deciding whether the candidate is pro-safety, anti-safety, pro-regulation, anti-regulation, or China/competition focused
	- if the source centrally frames AI in terms of China competition, national security, or export controls, include that framing in the description rather than flattening it into a generic AI label
	- if the source specifically supports guardrails, regulation, oversight, or research access, name that mechanism in the description instead of using a more generic AI phrase
	- if a source supports the claim but would be easier to verify with a better anchor, fragment, or deeper link, prefer updating the URL over deleting the row
	- if you replace an evidence URL with a better source, keep the core claim emphasis from the previous row only if the new source also clearly supports that emphasis; before proposing a URL replacement, inspect the replacement source itself and verify it directly supports the proposed description
	- do not add identity-only or candidacy-only evidence if stronger direct AI-related evidence is already present
	- if deleting a weak source because a better direct source exists, include the replacement in the same batch when possible
	- never delete the only remaining evidence row unless you also add a replacement source in the same batch that lets a voter verify the candidate's identity
- Status
	- `Status` is the descriptive human-readable race status (for example `Incumbent`, `Nominee`, `Dem Primary Candidate`, `Filed Candidate`, `Exploratory Candidate`)
	- only propose a `Status` change if the descriptive status itself is wrong or outdated
	- preserve useful descriptive detail in `Status`; do not collapse it to `In` or `Out`
	- the only allowed new `Status` value is `Out`
- Active
	- `Active` is the normalized voter-guide inclusion status and must be either `In` or `Out`
	- propose an `Active` change whenever the candidate should be included in or excluded from the voter guide
	- use `Out` for candidates who are no longer active in the race (for example removed from ballot, failed to qualify, exploratory-only, or clearly not running)
	- use `In` for candidates who are still active, filed, declared, competing in a primary/runoff/general, incumbent, challenger, or nominee
- Verdict
	- Modify if the balance of evidence (after revising) changes your opinion of their overall AI safety stance
	- do not change `Verdict` based on one weak or ambiguous source; prefer multiple direct signals when moving away from `no_record`
